<objective>
Create the color-validation utility (`src/utils/colorValidation.ts`) and apply load-time validation in `src/main.ts loadSettings()`. This addresses **SEC-01 success criterion #1** ("Loading a vault whose `data.json` contains a crafted color string ... does not inject markup — the value is replaced with the default color at `loadSettings` time") and is the prerequisite for plan 02-02 (render-time defensive guard via `createElementNS`).

Purpose: Stored color values are written to disk and re-read on load — they cross a trust boundary. A maliciously crafted `data.json` (e.g. from a synced vault, a shared template, or a downgrade attack) can contain a color string designed to break out of an SVG attribute context. Validating-and-replacing at load time is the structural fix; render-time `setAttribute` (plan 02-02) is belt-and-suspenders.

Create a new file `src/utils/colorValidation.ts`. The file has NO imports (browser globals only). Export three named functions:

**1. `isValidColor(value: string): boolean`** — Returns `true` iff the value matches one of the whitelisted CSS color formats below; `false` otherwise (including for `null`/`undefined`/empty string — defensive guard at the top).

The whitelist (per D-02) accepts these formats and ONLY these formats:
- `#rgb` (3-digit hex)
- `#rgba` (4-digit hex with alpha)
- `#rrggbb` (6-digit hex)
- `#rrggbbaa` (8-digit hex with alpha)
- `hsl(...)` — any contents inside the parens
- `hsla(...)`
- `rgb(...)`
- `rgba(...)`
- `var(--<identifier>)` where `<identifier>` matches `[a-zA-Z0-9_-]+`

Implementation as a single union regex. Use this exact regex (anchored start to end, case-insensitive on hex letters):

```typescript
const VALID_COLOR_REGEX =
  /^(#[0-9a-fA-F]{3}([0-9a-fA-F]([0-9a-fA-F]{2}([0-9a-fA-F]{2})?)?)?|hsla?\([^()<>]*\)|rgba?\([^()<>]*\)|var\(--[a-zA-Z0-9_-]+\))$/;
```

Notes on the regex:
- The hex alternation `#[0-9a-fA-F]{3}([0-9a-fA-F]([0-9a-fA-F]{2}([0-9a-fA-F]{2})?)?)?` matches exactly 3, 4, 6, or 8 hex digits after `#` (it does NOT match 5 or 7 — verify with mental trace: 3 base + (1 + (2 + (2)?)?)? gives {3, 4, 6, 8}).
- The `hsla?\(...\)` and `rgba?\(...\)` branches use `[^()<>]*` to disallow nested parens AND markup-breaking characters (`<` and `>`) inside the function. That structurally prevents `hsl(1)"><script>...` style injection.
- `var(--[a-zA-Z0-9_-]+)` is the strictest branch — only CSS custom-property names allowed.
- The leading `^` and trailing `$` anchors are critical — without them a crafted value like `#abc<script>` would match the prefix.

The function body:
```typescript
export function isValidColor(value: string | null | undefined): boolean {
  if (!value || typeof value !== "string") {
    return false;
  }
  return VALID_COLOR_REGEX.test(value);
}
```

```typescript
async loadSettings() {
  this.settings = Object.assign({}, DEFAULT_SETTINGS, await this.loadData());

  // SEC-01 (D-03, D-04): validate stored color values against the whitelist.
  // A maliciously crafted data.json can contain a color string designed to
  // break out of an SVG attribute context. Replace invalid values silently
  // and warn for diagnostics; no Notice (typical user can't act on it).
  this.settings.calendarUrls.forEach((source, index) => {
    if (source.color && !isValidColor(source.color)) {
      console.warn(
        `MemoChron: Invalid color "${source.color}" on calendar "${source.name}" — replacing with default.`
      );
      source.color = defaultColorForIndex(index);
    }
  });

  if (
    this.settings.dailyNoteColor &&
    !isValidColor(this.settings.dailyNoteColor)
  ) {
    console.warn(
      `MemoChron: Invalid dailyNoteColor "${this.settings.dailyNoteColor}" — replacing with default.`
    );
    this.settings.dailyNoteColor = defaultDailyNoteColor();
  }
}
```

Notes:
- The `if (source.color && ...)` guard is intentional — if `source.color` is `undefined` (older settings, or new sources before color assignment) we skip rather than overwriting with a default; that path is handled elsewhere in `SettingsTab` when the user opens settings.
- Same for `dailyNoteColor` — only validate if a value is present.
- Do NOT call `await this.saveData(...)` after the replacement. The corrected value will be written back the next time the user changes any setting (Obsidian's `saveSettings` flow). Forcing a save here would write the user's vault on every load and is out of scope.
- Per D-04: silently replace + `console.warn`. NO `new Notice(...)` — invalid colors are a developer-diagnostic event, not a user-actionable one.
- The `"MemoChron: "` prefix on the `console.warn` matches the established logging convention (CalendarService.ts:331,333).

Do NOT modify any other part of `main.ts`. Specifically:
- Do not change `onload`, `onunload`, `saveSettings`, `setupAutoRefresh`, or any service registration.
- Do not introduce `dailyNoteColor` defaults if it was undefined to begin with — only replace existing invalid values.
