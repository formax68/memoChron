This plan owns `src/services/CalendarService.ts` for Phase 2 and addresses TWO requirements:

1. **BUG-06** ‚Äî Replace the boolean `isFetchingCalendars` short-circuit at `src/services/CalendarService.ts:38, 42-48, 229, 250` with a **shared in-flight Promise** held on the service instance. Concurrent callers (force-refresh user click, `scheduleBackgroundRefresh` timer, embedded-view consumers) await the same Promise and receive the same result ‚Äî eliminating the double-render-or-duplicate-event-list race. Per ROADMAP success criterion #5 ("A second background-refresh triggered while a fetch is in flight does not produce a double-render or duplicate event list") and REQUIREMENTS.md BUG-06.

2. **SEC-02 (CalendarService.ts arm)** ‚Äî Normalize every non-discarded catch block in this file through `errorMessage(error)` from plan 02-03's `src/utils/errors.ts`, and rewrite the unsafe `error.message` accesses in the per-source fetch catch (lines 396, 398) and the local-file outer catch (line 532) that currently produce `[object Object]` user-facing Notices or trigger a secondary TypeError on non-Error throwables. Per ROADMAP success criterion #3 ("Every visible error in the plugin (failed fetch, failed note creation, failed cache read) shows a specific message rather than `[object Object]` or `undefined`") and REQUIREMENTS.md SEC-02.

Why both in one plan: Phase 2 makes substantive edits to `CalendarService.ts` in exactly two seams (the fetch-dedup wiring + the catches). Bundling them into one plan keeps the file's diff coherent and avoids a separate plan that would re-open the same file. The catch normalization runs first (Task 1) so the `errorMessage` import is present before the fetchInFlight task touches `performFetch`'s catch.

Purpose:
- BUG-06: the current boolean guard returns `this.events` (which may be stale or even `[]` if the in-flight fetch hasn't yet populated it) to the second caller. The shared-Promise pattern is the documented JS approach: hold the Promise on the instance, clear it in `finally`, and the Promise's non-null state IS the in-flight signal.
- SEC-02: the per-source fetch catch (~line 391) does `error.message && error.message.includes('CORS')` and the local-file outer catch (~line 528) does `\`Error reading file: ${error.message}\`` directly in the returned text. For non-Error throwables (which ical.js and the Obsidian requestUrl wrapper can produce), `error.message` is `undefined`, so the Notice / status text reads `undefined` and the `includes()` call throws a secondary TypeError. The `errorMessage(error)` helper from plan 02-03 returns a string unconditionally, eliminating both failures.
