<objective>
Plan numbering: 01-10 closes IN-01 from `.planning/phases/01-foundation/01-REVIEW.md`.

IN-01: `CalendarView.handleDragMoveBound` and `handleDragEndBound` (`src/views/CalendarView.ts:30-31`) are typed `(e: MouseEvent) => void` (non-nullable), but they are only assigned in `createUI()` which is called from `onOpen()`. If any code path accessed them before `onOpen()` completed (or if `onClose()` ran without `onOpen()` — e.g., a leaf that errored out during open), TypeScript's `strictNullChecks` would not catch the access because the declared type does not include `undefined`. The `isDragging` guard in `onClose()` (line 53) prevents an actual runtime access of the uninitialized fields, so there is no current bug — this is a latent type-safety gap.

Fix per REVIEW.md: declare the fields as `((e: MouseEvent) => void) | undefined` with explicit `= undefined` initializers, and add a defensive `&& this.handleDragMoveBound && this.handleDragEndBound` check inside the existing `onClose` guard so strictNullChecks is satisfied at the `removeEventListener` call sites.

D-09 invariant: onClose must NOT call handleDragEnd. The new guard does NOT change this — only the two `removeEventListener` calls remain inside the guard, and the `isDragging = false` clear after them is preserved. Only the conditional that wraps them is widened to also assert non-undefined-ness of the two handler fields.

Purpose: Strict-null-checks now reflects runtime reality. If a future refactor accidentally introduces a code path that touches these fields before `createUI()` runs, the TypeScript compiler will catch it.

Output: One file modified. One atomic commit.
</objective>

The defensive `&& this.handleDragMoveBound && this.handleDragEndBound` clauses narrow the type back to the function form inside the block.

D-09 constraint: the body inside the guard must remain exactly the two `removeEventListener` calls plus `this.isDragging = false`. Do NOT add a `handleDragEnd` call.

// handleDragEnd (line 1092-1093):

At those sites the fields are guaranteed assigned, so TypeScript's narrowing is via the prior assignment in handleDragStart/handleDragEnd's call chain.
