<objective>
Plan numbering: 01-06 closes both CR-01 (CRITICAL, plugin-level 100ms timer) and IN-02 (info-level, view-level 50ms timer). Merged into one plan because both findings have the same root cause (setTimeout handle passed to an API that internally calls clearInterval at unload) and the same fix pattern (own the handle, clear it with the matching window.clearTimeout in the matching teardown path).

Close CR-01 and IN-02 from `.planning/phases/01-foundation/01-REVIEW.md`. Both flag the same anti-pattern: a `setTimeout` handle is passed to `registerInterval`, which internally calls `clearInterval(id)` at unload. On Chromium/Electron the setTimeout/setInterval ID pools are shared and `clearInterval` cancels a setTimeout — so this works on desktop. On WKWebView (iOS) and Android WebView the spec does not require shared pools — `clearInterval` may silently fail to cancel the timeout, leaving a one-shot timer that fires against a partially torn-down service or view.

CR-01 (`CalendarService.scheduleBackgroundRefresh`, line 185-187) is the higher-risk site because plugin-level Component.unload() runs after service teardown — the documented residual iOS risk in 01-VERIFICATION.md and the pending iOS UAT in 01-HUMAN-UAT.md.

IN-02 (`CalendarView.onOpen`, line 68-70) is the lower-risk same-class-of-bug site — the View Component unloads synchronously on leaf close, so the cancellation window is narrower, but the API mismatch is identical.

Purpose: After this plan lands, the iOS UAT in 01-HUMAN-UAT.md should pass — both one-shot timers are cancelled with `window.clearTimeout` (the correct API for `setTimeout` handles) in the correct teardown path.

Output: Three modified files. CR-01 fix moves timer ownership from `CalendarService` (which has no Component) up to `MemoChron` (which does, and whose `onunload` already runs explicit cleanup). IN-02 fix keeps timer ownership inside `CalendarView` (which IS a Component, with its own `onClose`). One atomic commit.
