<objective>
Plan numbering: 01-07 closes WR-01 from `.planning/phases/01-foundation/01-REVIEW.md`.

WR-01: `MemoChron.setupAutoRefresh` (main.ts:163-171) calls `this.clearRefreshTimer()` (which cancels the old interval via `window.clearInterval`) and then calls `this.registerInterval(window.setInterval(...))` to create the new one. `Plugin.registerInterval` appends the new numeric ID to an internal array used for cleanup at unload. The old ID — already cancelled by `clearInterval` — is still in that array from the previous call. Each settings save leaves one stale numeric ID. `clearInterval` on a stale ID is a no-op (no crash), but the list grows without bound for the lifetime of the plugin if the user saves settings repeatedly.

Fix per REVIEW.md: drop the `registerInterval` wrap entirely. `onunload` already calls `clearRefreshTimer()` (line 93), which covers shutdown cleanup. The `registerInterval` was added in Plan 01-03 as belt-and-suspenders unload safety — REVIEW.md flags it as the cause of the unbounded growth, not a safety net.

Purpose: Eliminate the unbounded list growth without changing user-visible behavior. The auto-refresh interval still resets on every settings save (manual `clearRefreshTimer` + new `setInterval`) and is still cancelled at unload (explicit `clearRefreshTimer` in `onunload`).

Output: One file modified (`src/main.ts`). One atomic commit.
</objective>
