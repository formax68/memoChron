<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → Sigstore | Short-lived OIDC token is exchanged for a Sigstore signing certificate; the runner can only sign on behalf of `formax68/memoChron` |
| Release asset → end user (Obsidian / BRAT) | Users download `main.js`, `manifest.json`, `styles.css`; without attestation the artifact's provenance is unverifiable |
| GitHub attestations API → `gh attestation verify` (Plan 04, downstream users) | Persisted attestation entry is consulted by consumer-side verification |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing | Release artifact provenance (`main.js`, `manifest.json`, `styles.css`) | mitigate | `actions/attest-build-provenance@v3` produces a Sigstore-signed attestation in the public transparency log, keyed by the artifact's SHA-256 digest. Consumer check is `gh attestation verify` (Plan 04). [05-RESEARCH.md §6, §8] |
| T-05-02 | Spoofing / Tampering | Compromised CI runner forging attestation | mitigate | Short-lived OIDC token (~5 min lifetime); Sigstore SAN includes `SourceRepository`, `SourceRepositoryOwner`, and workflow ref — forging requires gaining the OIDC token AND impersonating the workflow path. Standard control, no custom code. [05-RESEARCH.md §"Security Domain"] |
| T-05-06 | Tampering | Attestation step ordering bug (attestation after release-create on immutable releases) | mitigate | Step ordering enforced: attestation runs AFTER build (so artifacts exist on disk) and BEFORE `gh release create` (sidesteps immutable-releases timing edge case, 05-RESEARCH.md §6.4 + §9.5 + actions/attest-build-provenance issue #734). Acceptance criterion above asserts `bi < ai < ci` in the YAML step list. |
| T-05-07 | Information Disclosure | Public attestation reveals build metadata | accept | Attestation metadata is intentionally public (provenance is the point). No secrets exposed; workflow path is already public in the repo. No mitigation needed. |
| T-05-03 (residual) | Tampering | Lint config or release workflow bypassed by malicious commit on `main` | accept | Branch-protection-rule on `main` requiring `lint.yml` / `release.yml` as required status checks is OUT OF SCOPE this phase per CONTEXT.md D-07 (repo-settings concern). The workflow itself is the mitigation; merge-time enforcement is a follow-up. |
</threat_model>
