<objective>
Implement ENH-02 (always-on agenda icon distinguishing events with notes from events without) and ENH-03 (optional, toggleable grid-cell indicator on days containing at least one event with a note). These two requirements travel together per STATE.md "Recent decisions" because they share a `RenderOptions` extension (D-06): a `hasNote(event)` callback that both renderers consume.

Per D-04, the agenda icon is always-on and rendered via Obsidian's `setIcon` API with the lucide names `file-check` and `file-plus` â€” these names are quoted verbatim by ROADMAP success criterion #2 and must not be substituted. The icon sits at the trailing end of each `.memochron-agenda-event` row, after the time/title/location block.

Per D-05, the grid indicator is toggled by a new boolean setting `showNoteIndicatorOnGrid`, default `false`. Visual: a small accent-colored filled square (`border-radius: 1px`, NOT round) positioned in a corner of the cell, distinct from the round event-color dots in `.memochron-event-dots-container` so the two layers do not visually merge.

Per D-06, the seam between renderers and the note-existence check is a callback on `RenderOptions` rather than a precomputed map. The callback shape is chosen because `CalendarEvent` lacks a stable identity key across renders (timezone-converted recurrences regenerate). The callback wraps `noteService.getExistingEventNote(event) !== null` â€” already cheap and synchronous (`vault.getAbstractFileByPath`).

Per D-07, the refresh model is re-render-after-create: `showEventDetails` triggers `renderCalendar()` and `showDayAgenda()` after `createEventNote` returns successfully. No vault event listener is added (out of scope per D-07).

**Cross-surface refresh parity (addresses checker warning #2):** D-07 originally scoped only to `CalendarView.showEventDetails` on the sidebar surface. To honor the cross-surface consistency precedent established by ENH-01 D-02 (which deliberately reaches both surfaces via a single CSS class), Task 3 extends the post-create re-render to `EmbeddedAgendaView.handleEventClick` by calling `this.render()` after `await leaf.openFile(file)` in the new-note branch. Without this, the embedded agenda surface would show a stale `file-plus` icon until the user manually reloads the host note â€” a UX divergence shipped users would file as a bug.

Per the per-surface-duplication observation in PATTERNS.md "Established Patterns": both the shared renderers in `src/utils/viewRenderers.ts` AND the sidebar's private methods in `src/views/CalendarView.ts` need touching. Embedded views go through the shared renderers; the sidebar has its own private `renderEventItem` (line 827) and `addDayEventIndicator` (line 629) that mirror the shared functions.

This addresses **ROADMAP Phase 4 success criteria #2 and #3**.

Purpose: make note-state visible without opening the note. Agenda always shows the per-event state; calendar grid optionally shows per-day aggregate state for users who want it. The same icon-flip behavior happens on both sidebar and embedded surfaces.
