<objective>
Implement ENH-06: a `{{cursor}}` marker in a note template places the editor cursor at that position after a newly-created event note opens. The marker text never appears in the saved note.

Per D-12, marker stripping is unconditional: after template substitution but before `vault.create`, every `{{cursor}}` occurrence is removed from the content that will be written. Even if cursor placement subsequently fails (no active editor, view type mismatch), the saved file is clean — ROADMAP success criterion #6 holds unconditionally.

Per D-13, the marker target is the first `{{cursor}}` AFTER the closing `---` frontmatter delimiter. Markers inside frontmatter are stripped (so they don't corrupt the YAML) but NOT honored as cursor targets — REQUIREMENTS Out of Scope explicitly anti-features "Honoring `{{cursor}}` inside YAML frontmatter — would corrupt frontmatter."

Per D-14, cursor placement runs only when `isNewNote === true`. If a user later types the literal string `{{cursor}}` into a saved note, opening that note later must NOT move their cursor — the marker is a creation-time directive, never a file-content directive.

Per D-15, the timing mechanism after `await leaf.openFile(file)` is ONE `requestAnimationFrame` yield, then `getActiveViewOfType(MarkdownView)` → `editor.setCursor(pos)` → `editor.focus()`. No retry loop, no `setTimeout` fallback. PERF-04 (the broader timeout-sweep) is deferred to a future perf milestone.

**Cross-surface scope (addresses checker blocker #1):** `createEventNote` has **THREE** call sites in the repo, not two. Verified with `grep -rn 'createEventNote' src/`:
1. Definition in `src/services/NoteService.ts:62` (and its internal callers)
2. Sidebar consumer at `src/views/CalendarView.ts:925` (inside `showEventDetails`)
3. **Embedded agenda consumer at `src/views/EmbeddedAgendaView.ts:393` (inside `handleEventClick`)**

The return-shape change from `Promise<TFile>` to `Promise<{ file: TFile; cursor: ... }>` will break compilation at all consuming call sites. The original revision of this plan missed call site #3, which would cause `npx tsc -noEmit` to fail with a type error like `Property 'basename' does not exist on type '{ file: TFile; cursor: ... }'`. Task 2 below covers **both** the sidebar (`showEventDetails`) and the embedded agenda (`handleEventClick`) destructuring + cursor-placement updates, mirroring the ENH-02/03 D-07 cross-surface extension precedent already shipped in plan 04-04.

This is the highest-risk plan in Phase 4 per STATE.md Blockers/Concerns: "Phase 4 (ENH-06): Confirm exact tick mechanism for `setCursor` timing at implementation time (`requestAnimationFrame` vs `app.workspace.onLayoutReady`) — do not bake in an assumption." The decision in D-15 is `requestAnimationFrame`. If implementation reveals `requestAnimationFrame` is insufficient (editor still null on first frame), the fallback is documented in the plan summary; if a fallback is needed, plan 04-05 SUMMARY records the chosen mechanism and a one-line rationale.

This addresses **ROADMAP Phase 4 success criterion #6**.

Purpose: respect the user's template intent for cursor placement on BOTH surfaces, without ever leaving marker text in the saved file.

|-----------|----------|-----------|-------------|-----------------|
| T-04-12 | Tampering | `{{cursor}}` marker stripping | mitigate | Unconditional strip-before-write: the `extractCursorMarker` helper removes ALL `{{cursor}}` occurrences from the content passed to `vault.create`, regardless of marker count or position. Per D-12, ROADMAP success criterion #6 ("the marker text itself does not appear in the saved note") holds even if cursor placement fails. Verified by the Task 1 REPL test cases and the Task 2 end-to-end smoke tests (reading the saved file from disk after creation, on BOTH surfaces). |
| T-04-13 | Tampering | YAML frontmatter integrity | mitigate | Per D-13, markers inside frontmatter are stripped but NOT honored as cursor targets. The frontmatter is well-formed before substitution (`generateFrontmatter` at lines 176-199); after substitution it can contain a `{{cursor}}` if the user's template puts one there. Stripping the marker from frontmatter removes a substring that would otherwise sit inside a YAML value (or worse, between keys, breaking parser). The strip pass uses `split(MARKER).join("")` — a literal-string operation, no regex backtracking risk. Verified by Task 2's "frontmatter-only marker" smoke test. |
| T-04-14 | Tampering | Cursor placement on existing notes | mitigate | Per D-14, `setCursor` runs only when `isNewNote === true` on BOTH surfaces. A user who later types the literal string `{{cursor}}` into a saved note will NOT have their cursor moved on re-open — the marker is a creation-time directive, not a file-content directive. Verified by Task 2's "existing-note re-open" smoke test on both sidebar and embedded surfaces. |
| T-04-15 | Denial of Service | `extractCursorMarker` on a large template body | accept | The helper performs two passes over the content string: one O(n) line split + O(n) marker search, one O(n) `split + join`. Total O(n) for content of size n. Even a 100KB template is processed in single-digit milliseconds. No regex backtracking; no recursion. |
| T-04-16 | Denial of Service | `requestAnimationFrame` callback fires after view destruction | accept | If the user closes the just-opened tab before the next frame, `getActiveViewOfType(MarkdownView)` returns `null` or a different view, `view?.editor` evaluates falsy, and the callback aborts silently. No exception, no error notice. Applies to both surfaces. |
| T-04-17 | Elevation of Privilege | `setCursor` and `editor.focus()` calls | accept | `MarkdownView.editor.setCursor` and `editor.focus()` are Obsidian-provided APIs that operate within the editor instance. They do not modify file content, vault structure, or plugin settings. No privilege boundary is crossed. |

**Conclusion:** Three threats mitigated by explicit design decisions (D-12, D-13, D-14); three accepted as low-risk by analysis. No `high`-severity findings; nothing to block on.
