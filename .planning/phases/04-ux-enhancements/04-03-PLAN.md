<objective>
Implement ENH-05: support `{{day}}` and `{{month}}` template variables that emit fully-written English weekday and month names (`Monday`, `January`). Per D-09, locale is hard-coded `"en-US"` because ROADMAP success criterion #5 explicitly names the **English** weekday/month name as the acceptance signal; locale customization (NOTE-01) is REQUIREMENTS Out of Scope.

Per D-10, both variables flow into note body AND note title automatically because `applyTemplateVariables` (around `src/services/NoteService.ts:254-272`) iterates the variables map for both call sites — `generateNoteContent` body (line 162) and `formatTitle` title (lines 274-280). The implementation is therefore three small edits:

1. Add `day: string` and `month: string` to the `EventTemplateVariables` interface (lines 7-27)
2. Populate the two fields in `getEventTemplateVariables` (around line 218) via `event.start.toLocaleDateString("en-US", { weekday: "long" })` and `{ month: "long" }`
3. Add `{{day}}` and `{{month}}` to the help text near the template-input UI in `SettingsTab.ts` so users can discover the new variables

Per D-11, source date is `event.start` — consistent with the other date/time variables. No short-form variants this phase (would be scope creep per D-11). The values `Monday` / `January` are filename-safe (only `[A-Za-z]`) so no extra sanitization is needed beyond what already exists.

This addresses **ROADMAP Phase 4 success criterion #5**.

Purpose: smallest-surface template extension. The variables flow through the same iterator already used by every existing variable; no new substitution machinery is created.

- `src/settings/SettingsTab.ts` — help text near the template field updated to list `{{day}}` and `{{month}}`

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | New variable values flowing into note title (filename) | accept | `toLocaleDateString("en-US", { weekday: "long" })` produces values from a fixed set of seven strings (`Sunday`-`Saturday`). `toLocaleDateString("en-US", { month: "long" })` produces values from a fixed set of twelve strings (`January`-`December`). All 19 possible values match `[A-Za-z]+` only and are unaffected by the existing `sanitizeFileName` regex (`[\\/:*?"<>|]` replaced with `-`). No filesystem-path injection possible. |
| T-04-06 | Tampering | New variable values flowing into note body (markdown) | accept | The values are inserted by `applyTemplateVariables` as plain text (via `replaceAll`) into a string that is written to a vault file by `vault.create`. Obsidian renders the saved file as markdown; the values contain no markdown metacharacters (no `*`, `_`, `[`, `#`, etc.) so no unintended formatting can be triggered. |
| T-04-07 | Information disclosure | Hard-coded `"en-US"` locale string | accept | Per D-09 the `"en-US"` locale is intentional — ROADMAP success criterion #5 names the English weekday/month name as the required output. The hard-code is documented in CONTEXT.md and visible in source. |

**Conclusion:** No new attack surface. All threats are LOW; nothing to block on.
