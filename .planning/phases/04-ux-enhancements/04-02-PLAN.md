<objective>
Implement ENH-04: surface the existing `DD-MM-YYYY` date format option to NL/EU users by relabeling the dropdown entry. The format is already implemented — `NoteService.formatDate` (around `src/services/NoteService.ts:340-347`) already handles `"UK"` via `toLocaleDateString("en-GB", { day: "2-digit", month: "2-digit", year: "numeric" })` and routes through `toFilenameSafeDate` to produce `15-01-2026`-style filenames. The only thing missing is discoverability: NL users searching the dropdown for "DD-MM-YYYY" or "EU" don't see a match.

Per D-08, the implementation is a **label-only change in two dropdown definitions**:
- Global setting at `src/settings/SettingsTab.ts:937-943` (inside `renderNoteDateFormat`)
- Per-calendar override at `src/settings/SettingsTab.ts:1584-1590`

The persisted `value: "UK"` MUST remain unchanged in both arrays so existing user `data.json` files keep working without migration code. Per CONTEXT.md "Claude's Discretion": chosen approach is to **relabel the existing entry** rather than add a parallel `"EU"` value entry — keeps the dropdown short, avoids two synonymous values resolving to the same formatter, and is the simplest change that satisfies the requirement.

This addresses **ROADMAP Phase 4 success criterion #4**.

Purpose: zero-risk label edit on a working formatter. The label change is the entire fix.

Output:
- `src/settings/SettingsTab.ts` — two string literals updated from `"UK (DD-MM-YYYY)"` to `"UK/EU (DD-MM-YYYY)"`
- No change to `NoteService.formatDate` (already covers the format)
- No new constants or types
- One atomic commit for ENH-04
</objective>
